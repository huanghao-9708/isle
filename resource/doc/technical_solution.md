# 「私屿」Dioxus技术方案

本文档为「私屿」本地音乐播放器项目的技术落地方案，基于 Dioxus 跨平台框架和 Cargo Workspace 架构设计，详细阐述架构设计、技术选型、核心模块实现与性能优化策略，完全匹配项目「数据绝对私有化、跨平台兼容、轻量流畅」的核心定位。

---

## 一、方案概述

### 1.1 项目背景

「私屿」是一款主打隐私安全的本地音乐播放器，核心目标是为拥有本地音乐库的用户提供**零数据上报、跨平台无缝同步、高颜值轻量**的播放与管理体验。项目采用 Cargo Workspace 组织架构，实现一套代码多端运行，同时满足极致性能与隐私安全的要求。

### 1.2 选择 Dioxus + Cargo Workspace 的核心优势

针对本项目的需求，Dioxus + Cargo Workspace 架构相比其他方案具备以下不可替代的优势：

1. **极致轻量**：基于 Rust 的原生编译，打包后应用体积小（基础应用仅～5MB），资源占用远低于 Electron，冷启动速度可控制在 2 秒内，完全匹配项目的轻量定位。

2. **全平台统一**：通过 Cargo Workspace 组织代码，共享 UI 组件，分别为 Web、Desktop、Mobile 编译，完美实现跨平台统一的 UI 与交互体验，解决现有播放器跨平台体验割裂的问题。

3. **原生性能**：Rust 的内存安全与零成本抽象，保证音频处理、大容量文件扫描、加密同步等 heavy 任务的性能，可轻松支撑 TB 级音乐库的流畅运行。

4. **隐私友好**：原生编译无运行时，可完全控制所有网络行为，天然支持零数据上报的隐私需求，所有代码可审计，符合项目的隐私安全定位。

5. **开发效率**：借鉴 React 的组件化与 Hook 模型，开发体验友好，组件复用性高，可快速迭代 MVP 与后续功能。

6. **代码复用**：通过 `ui` 包共享组件，`api` 包共享核心逻辑，各平台包（web/desktop/mobile）仅需处理平台特定逻辑，最大程度复用代码。

## 二、核心技术选型

|技术领域|选型方案|选型说明|
|---|---|---|
|**UI 框架**|Dioxus 0.7+daisyUI|跨平台 UI 核心，提供组件化开发、响应式状态管理，支持多平台渲染|
|**项目组织**|Cargo Workspace|通过 Workspace 组织多个包，实现代码复用和依赖统一管理|
|**核心逻辑**|api 包|所有业务逻辑、数据层、基础设施层集中在 api 包中|
|**音频处理**|Symphonia + Rodio|Symphonia 负责全格式音频解码（支持 FLAC、APE、WAV、MP3、M4A 等），Rodio 负责跨平台音频输出|
|**本地数据存储**|SQLite|经典高可用的数据库|
|**状态管理**|Dioxus Signal + Global Signal|基于 Dioxus 原生的响应式状态系统，组件内状态用`use_signal`，全局共享状态用全局 Signal|
|**网络与同步**|Reqwest + Webdav-Client|异步 HTTP 客户端，支持 WebDAV 协议，实现与用户私有云 / NAS 的对接|
|**加密安全**|AES-GCM + RustCrypto|端到端加密同步数据，采用 AES-GCM 对称加密|
|**UI 样式**|Tailwind CSS v3 + DaisyUI v4|原子化 CSS 框架配合 DaisyUI 组件库，快速实现统一的高颜值 UI|
|**大数据渲染**|Dioxus Virtual List|自定义虚拟列表组件，仅渲染可视区域的列表项|
|**打包构建**|Dioxus CLI|官方构建工具，一键打包多平台安装包|

### 3.4 模块交互流程

以「添加音乐文件夹并扫描」为例，模块间的交互流程如下：

1. UI 层（web/desktop/mobile）：用户在设置页面选择本地文件夹，调用 `api::library::scan_folder`

2. 业务层（api 包 services/）：音乐库服务接收任务，提交到后台线程池，避免阻塞 UI

3. 基础设施层（api 包 infrastructure/）：文件系统模块遍历文件夹，识别音频文件

4. 业务层（api 包 services/）：对每个文件，调用音频引擎的元数据解析接口，提取歌曲信息

5. 数据层（api 包 data/）：将解析后的元数据持久化到 SQLite 数据库，更新索引

6. 业务层（api 包 services/）：实时推送扫描进度到 UI 层

7. UI 层（web/desktop/mobile）：接收进度更新，刷新界面，扫描完成后刷新音乐库列表

## 四、核心模块详细设计

### 4.1 核心逻辑（api 包）

`api` 包是整个项目的核心，所有平台共享这套逻辑：

#### 4.1.1 基础设施层（api/src/infrastructure/）

最底层的通用能力封装，向上提供统一的抽象接口，屏蔽不同平台的差异：

- **音频引擎**：封装跨平台的音频解码与播放能力，向上提供统一的播放控制接口
- **文件系统抽象**：封装不同平台的文件访问 API，处理不同平台的存储权限差异
- **网络层**：封装异步 HTTP 请求，提供 WebDAV 客户端能力
- **加密工具**：封装端到端加密、哈希计算等通用安全工具
- **平台适配**：处理不同平台的系统 API 调用，比如通知、快捷键、蓝牙设备等

#### 4.1.2 数据层（api/src/data/）

负责数据的持久化、建模与同步，是整个应用的数据核心：

- **数据模型**：定义强类型的核心数据结构，比如`Track`（歌曲元数据）、`Playlist`（播放列表）、`SyncStatus`（同步状态）等
- **本地存储**：基于 SQLite 实现数据的持久化存储，提供 CRUD 接口，处理数据的版本兼容
- **缓存管理**：对热点数据（比如最近播放的歌曲、专辑封面）做内存缓存，提升访问速度
- **数据同步**：处理本地数据与云端同步数据的合并、冲突解决

#### 4.1.3 业务逻辑层（api/src/services/）

封装核心业务能力，向上层 UI 提供可调用的服务接口，是业务规则的核心：

- **播放服务**：封装播放控制、播放队列、循环 / 随机播放等核心播放逻辑
- **音乐库服务**：封装音乐文件扫描、元数据解析、封面匹配、音乐库索引管理
- **同步服务**：封装私有云同步逻辑，处理同步任务的调度、加密、断点续传、冲突处理
- **标签与档案服务**：封装自定义标签、私人音乐档案的管理逻辑

#### 4.1.4 全局状态管理（api/src/state/）

基于 Dioxus 的响应式状态系统，实现全局状态的统一管理：

- **播放状态**：当前播放的歌曲、播放状态、进度、音量等，全局共享，所有组件可订阅
- **音乐库状态**：音乐库的加载状态、扫描进度、筛选条件等
- **同步状态**：同步任务的状态、进度、错误信息等
- **应用配置**：用户的设置、主题、同步配置等，持久化到本地

### 4.2 共享 UI 组件（ui 包）

`ui` 包是整个项目的 UI 核心，所有平台共享这套组件：

#### 核心组件

1. **MusicLibrary** - 音乐库主组件
   - 顶部导航栏
   - 统计看板
   - 歌曲列表表格
   - 底部播放控制栏

2. **通用组件**
   - Hero - 首页英雄区
   - Navbar - 导航栏
   - Echo - 测试组件

3. **样式资源**
   - Tailwind CSS v3 + DaisyUI v4
   - 统一的主题配置
   - 共享的 CSS 资源

### 4.3 播放引擎模块（api 包）

核心负责音频的解码、输出与播放控制，是播放器的基础能力：

#### 核心能力
- **全格式解码**：基于 Symphonia 实现对 FLAC、APE、WAV、MP3、M4A、OGG 等主流音频格式的支持
- **跨平台输出**：基于 Rodio 实现跨平台的音频输出
- **播放控制**：播放/暂停、切歌、进度跳转、音量控制
- **播放队列管理**：顺序/循环/随机播放模式

#### 架构设计
播放引擎采用独立线程运行，避免阻塞 UI 线程。

### 4.4 音乐库管理模块（api 包）

负责本地音乐文件的扫描、元数据管理与索引：

#### 核心能力
- **增量扫描**：首次全量扫描，后续仅检查修改时间
- **元数据自动匹配**：解析音频文件内置元数据
- **高效索引**：基于 SQLite 的索引能力
- **封面墙渲染**：基于虚拟列表实现

### 4.5 私有云同步模块（api 包）

核心实现用户私有云的端到端加密同步：

#### 核心能力
- **WebDAV 通用兼容**：支持标准 WebDAV 协议
- **端到端加密**：AES-GCM 对称加密
- **元数据同步**：MVP 阶段仅同步轻量元数据
- **冲突解决**：多设备同步冲突处理
- **断点续传**：支持同步任务断点续传

```bash
curl -sSL http://dioxus.dev/install.sh | sh
```

#### 开发 Web 平台
```bash

npm run watch:css

# 终端 2 - 启动 Web 开发服务器
cd packages/web
dx serve
```

```bash

npm run watch:css

# 终端 2 - 启动 Desktop 应用
```

```bash
# 构建 CSS
npm run build:css

# 构建 Web
cd packages/web
dx build --release

# 构建 Desktop
cd packages/desktop
dx build --release --platform desktop
```

3. 在任意平台包中使用该组件

## 六、隐私安全技术保障

### 6.1 绝对零数据上报
- **编译级保障**：核心编译产物中完全移除任何数据上报代码
- **网络行为审计**：所有网络请求仅在用户主动触发同步时才会发起
- **开源可审计**：所有核心代码开源

### 6.2 端到端加密同步
- **数据全程加密**：同步数据从本地加密后再上传
- **密钥用户掌控**：加密密钥完全由用户保管
- **无中心化服务**：不搭建任何中心化的云服务

### 6.3 完全离线可用
- **核心功能离线**：播放、音乐库管理 100% 支持离线
- **无联网授权**：无需联网激活、无需登录账号
- **本地数据存储**：所有用户数据默认存储在本地

## 七、性能优化方案

### 7.1 启动速度优化
- **按需加载**：启动时仅加载核心功能
- **预编译资源**：UI 资源预编译到二进制文件
- **冷启动优化**：桌面端控制在 2 秒内

### 7.2 大容量音乐库优化
- **虚拟列表**：所有列表采用虚拟列表
- **增量索引**：索引增量更新
- **后台任务优先级**：低优先级线程处理后台任务

### 7.3 渲染性能优化
- **Memo 缓存**：复杂计算使用 `use_memo` 缓存
- **按需渲染**：非活跃页面懒加载
- **细粒度状态更新**：基于 Signal 仅更新变化的组件

## 十、总结

本方案基于 Dioxus 0.7 框架和 Cargo Workspace 架构，为「私屿」本地音乐播放器设计了一套完整的跨平台技术架构。通过 Workspace 组织多个包，将**所有核心逻辑集中在 api 包中**，UI 组件集中在 ui 包中，各平台包仅需处理平台特定逻辑，最大程度复用代码。这种架构既保证了 MVP 的快速落地，也为后续的功能迭代与扩展预留了足够的空间。

该方案可以实现一套代码多端运行，同时从技术层面保障了用户数据的绝对私有化，填补了市场上的空白，为用户提供真正纯净、自主的私人音乐管理体验。
